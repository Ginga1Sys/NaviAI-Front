# 公開トップ画面作成 調査結果

## 概要
- 公開トップ画面（SCR-12）で想定している「未ログインで公開記事2件を表示」「タグ一覧を表示」は、現状実装のままでは実現できない。
- 理由は、バックエンドのセキュリティ設定で `/api/v1/auth/**` と `/h2-console/**` 以外の全 API が認証必須になっており、`GET /api/v1/knowledge` と `GET /api/v1/tags` も未ログインでは呼び出せないためである。
- さらに、`knowledge` テーブルと `Knowledge` エンティティには「公開トップで外部公開可能か」を判定する項目が存在しない。現状の `status='published'` は承認済み・公開済みの状態を示すだけで、未ログイン公開可否を区別できない。
- 公開トップ画面向けには、既存の認証必須 API をそのまま開放するのではなく、公開対象だけを返す専用の read API を追加する方針が適切である。

## 前提条件
- AIナレッジ共有サイトのフロントエンドは Next.js（React / TypeScript）、バックエンドは Spring Boot で実装されている。
- 公開トップ画面は未ログイン利用者向け画面であり、ログイン導線と会員登録導線を持つ。
- 既存のダッシュボード画面では `GET /api/v1/knowledge` を利用して「おすすめ1件」「新着3件」を取得している。
- 公開トップ画面の既存資産は `NaviAI-Front/app/pages/public-home/index.tsx` に存在するが、現状はダミーデータ表示であり API 連携は未実装である。

## 調査内容と結果

### 1. `GET /api/v1/knowledge` は未ログイン状態で呼び出せるか
結論として、現状は呼び出せない。

根拠は以下の通り。

- `SecurityConfig` では `/api/v1/auth/**` と `/h2-console/**` のみ `permitAll()` で、それ以外は `anyRequest().authenticated()` になっている。
- `KnowledgeController` でも、`GET /api/v1/knowledge` は通常の認証必須 API として実装されている。
- `KnowledgeServiceImpl` は取得条件として `k.is_deleted = false AND k.status = 'published'` を付与しているため、返却対象は「published の記事」に限定されているが、これは未ログイン公開可否の判定ではない。

補足:

- 基本設計書（API）では `GET /api/v1/knowledge` は「Auth: optional（公開/非公開ポリシーに準拠）」と記載されているが、現行コードはその設計に追従していない。
- ダッシュボード実装はこの API を認証付きで利用しているため、安易に匿名開放すると既存画面の情報公開範囲に影響する。

### 2. `GET /api/v1/tags` は未ログイン状態で呼び出せるか
結論として、現状は呼び出せない。

根拠は以下の通り。

- `TagController` 自体は単純な `GET /api/v1/tags` エンドポイントだが、セキュリティ設定上は `anyRequest().authenticated()` の対象である。
- `TagServiceImpl` はタグ利用数をそのまま返しており、公開記事に紐づくタグだけに限定する処理は存在しない。

補足:

- もし既存の `GET /api/v1/tags` を匿名開放すると、公開トップ画面に不要な内部タグまで露出する可能性がある。
- 公開画面で表示するタグは「公開対象記事に付与されたタグのみ」に絞るべきである。

### 3. 既存 API を未ログインでも呼べるようにするべきか、公開トップ画面用 API を新設するべきか
推奨は「公開トップ画面用の read API を新設する」方針である。

推奨理由は以下の通り。

- `GET /api/v1/knowledge` はダッシュボード、検索結果一覧、クイック検索など複数のログイン後画面で使われており、既存の業務向け仕様と公開向け仕様が混在すると条件分岐が複雑になる。
- 未ログイン公開では「公開可の記事のみ」「公開可の記事から集計したタグのみ」という制約が必要であり、既存 API をそのまま開放すると情報漏えいリスクが高い。
- セキュリティ設定でも公開用エンドポイントを個別に `permitAll()` した方が、公開範囲をコード上で明確に管理しやすい。

推奨する API 例:

- `GET /api/v1/public/knowledge?filter=latest&size=2`
  - 公開トップ用の公開記事一覧を返す。
  - 未ログインでも利用可能。
  - 条件は最低でも `status = 'published'` かつ `visibility = 'public'` を強制する。
- `GET /api/v1/public/knowledge?filter=recommended&size=1`
  - 仕様上「おすすめ記事（ログインが必要）」を公開トップにどう見せるかで分岐する。
  - 本当にログインが必要な記事を表示するなら、未ログイン画面には実データを返さず、固定メッセージまたはログイン誘導カードとして扱う方が安全である。
- `GET /api/v1/public/tags`
  - 公開記事に付与されたタグのみを集計して返す。

代替案として、既存の `GET /api/v1/knowledge` と `GET /api/v1/tags` を匿名利用可能にし、未ログイン時だけ公開条件をサービス側で強制する構成も可能ではある。ただし以下の理由で優先度は低い。

- 認証有無でレスポンスの意味が変わるため、保守性が落ちる。
- 将来の検索条件追加時に「匿名時に許可してよい条件か」を毎回精査する必要がある。
- 内部向け published 記事やタグを誤って露出する事故が起きやすい。

### 4. 公開記事と非公開記事の判定はどう実装するべきか
現状のデータモデルでは判定できないため、`knowledge` テーブルに公開範囲を表すカラムを追加するのが適切である。

現状確認:

- `knowledge` テーブルには `status`, `is_deleted`, `published_at` はあるが、公開トップでの公開可否を表すカラムは存在しない。
- `Knowledge` エンティティにも同様の属性は存在しない。
- `status` は `draft / pending / published / declined / archived` のワークフロー状態であり、アクセス公開範囲とは責務が異なる。

推奨案:

- `knowledge` テーブルに `visibility` カラムを追加する。
- 例: `visibility VARCHAR(20) NOT NULL DEFAULT 'internal'`
- 想定値:
  - `internal`: ログインユーザー向けのみ表示
  - `public`: 未ログインでも表示可能

この方式を推奨する理由:

- 記事単位の単純な公開制御であり、新規テーブルを追加するほどの関係性は不要である。
- `status` と `visibility` を分離することで、「承認済みだが社内限定」「承認済みかつ公開トップ掲載可」を明確に区別できる。
- 将来的に `restricted` などの値を追加しやすく、`is_public BOOLEAN` より拡張しやすい。

公開条件の推奨定義:

- 未ログイン公開対象 = `status = 'published' AND visibility = 'public' AND is_deleted = false`

注意点:

- 公開タグ集計も、上記条件を満たす記事に紐づくタグだけを対象にする必要がある。
- 今後、公開記事詳細画面を未ログインで開けるようにする場合は、一覧 API だけでなく詳細 API 側にも同じ条件を適用する必要がある。
- 現状コード上では `GET /api/v1/knowledge/{id}` の実装が確認できていないため、公開記事遷移先の API は別途実装または調査が必要である。

### 5. フロントエンドへの影響
現状の公開トップ資産は API 未接続の静的モックであり、実装時には以下の考慮が必要である。

- `NaviAI-Front/app/pages/public-home/index.tsx` はダミー記事2件、タグ3件を直接描画している。
- プロジェクトのディレクトリ構成資料には「新規実装は `app/` 側を優先」とあるため、実装時は App Router 側への整理も検討した方がよい。
- `fetcher.ts` には `skipAuth` オプションが既にあり、未ログイン画面から認証ヘッダー無しで公開 API を呼び出す準備はできている。

実装時の推奨:

- 公開トップ画面は `skipAuth: true` で公開 API を呼ぶ。
- おすすめ記事が「ログインしないと閲覧不可」の要件である場合、未ログイン画面では実データ取得ではなく CTA カードとして表現する。
- もし「おすすめ1件も公開できる記事から選ぶ」運用に変更するなら、`/api/v1/public/knowledge?filter=recommended&size=1` を追加して公開記事から推薦順で1件返す構成が可能である。

## 最終結論
- 現状の `GET /api/v1/knowledge` と `GET /api/v1/tags` は未ログインでは利用できない。
- 公開トップ画面を安全に実装するには、既存 API の匿名開放ではなく、公開対象だけを返す専用 API を追加する方が適切である。
- 公開/非公開の判定は `status` だけでは不足しているため、`knowledge` テーブルへ `visibility` などの公開範囲カラムを追加する必要がある。
- フロントエンド側は既存の公開トップ資産を流用できるが、実データ取得に合わせて API 連携と未ログイン向け導線の見直しが必要である。

# 公開トップ画面（SCR-12）修正 作業手順書

## 概要
- 公開トップ画面の「今週の注目」表示をバックエンドのAPIから取得したおすすめ記事（いいね数上位）に差し替える。\
- 公開記事用のテストデータを3件追加し、そのうち2件が画面に表示されることを確認する。\
- 画面右上の「社内メンバー限定の機能」セクションを削除する。\

## 前提条件
- ローカルにリポジトリがクローン済みで、Docker が利用可能であること。\
- 開発用の docker-compose 構成は `NaviAI-Front/docker-compose.dev.yml` にあること。\
- テスト用データの格納先は `NaviAI-Back/app/src/main/resources/data.sql` を利用すること。

## 成果物（想定）
- フロントエンド: 公開トップ画面の `今週の注目` がAPIから取得したタイトルを表示するように修正されたコンポーネント（例: `NaviAI-Front/app/public-home` 配下の該当ファイル）。\
- フロントエンド: 右上の会員訴求セクションが削除済み。\
- バックエンド: 公開記事のテストデータ3件を `data.sql` に追加。\
- ドキュメント: ディレクトリ構成ファイルを更新（Front/Back）。\

## 実装方針（短く）
1. まずバックエンドに公開記事のテストデータを追加する（3件）。\
2. API が公開記事をいいね数で降順に返すことを確認、必要なら軽微なAPI修正を行う。\
3. フロントエンドでは静的タイトルを削除し、ヘッダのおすすめはAPIで取得したトップ1件を表示し、中央の新着は公開APIで上位2件を表示する。\
4. 右上の会員訴求ブロックを削除。\
5. 動作確認後にドキュメントを更新し、docker-composeで統合起動して最終確認。

## 詳細な実装手順

### 1) 資料確認
- 参照: [NaviAI-Front/docs/nishizawa/公開トップ画面作成/公開トップ画面_修正内容.md](NaviAI-Front/docs/nishizawa/公開トップ画面作成/公開トップ画面_修正内容.md) を読み、画面設計と要件を再確認する。

### 2) バックエンド：テストデータを追加（ローカル）
1. `NaviAI-Back/app/src/main/resources/data.sql` をバックアップする。\
2. テスト用の公開記事を3件追加する（非公開の記事のテストデータは既に用意されているため追加不要）。スキーマ差異があるため下記は例（プロジェクトの `articles` テーブル定義に合わせて調整する）。

SQL 例（必要に合わせてカラム名を修正してください）:

```sql
-- 例: 3件の公開記事（テスト用）
-- プロジェクトのスキーマでは記事テーブルが `knowledge` で、いいねは別テーブル("like")で管理されている可能性があります。
-- 以下はより現実的なシード例（必要に応じてカラム名をプロジェクト実装に合わせて調整してください）。

INSERT INTO knowledge (id, title, content, visibility, status, is_deleted, created_at, updated_at, author_id) VALUES
(1001, '公開テスト記事 A', 'テスト用コンテンツA', 'public', 'published', false, NOW(), NOW(), 1),
(1002, '公開テスト記事 B', 'テスト用コンテンツB', 'public', 'published', false, NOW(), NOW(), 1),
(1003, '公開テスト記事 C', 'テスト用コンテンツC', 'public', 'published', false, NOW(), NOW(), 1);

-- いいね数は集計で求めるケースが多いので、いいねを別テーブルに挿入して疑似的に件数を作ります。
-- 例: id 2001.. を使って knowledge_id=1001 に 3 件、1002 に 2 件、1003 に 1 件のいいねを作る
INSERT INTO "like" (id, user_id, knowledge_id, created_at) VALUES
(2001, 2, 1001, NOW()),
(2002, 3, 1001, NOW()),
(2003, 4, 1001, NOW()),
(2004, 2, 1002, NOW()),
(2005, 3, 1002, NOW()),
(2006, 2, 1003, NOW());

-- 注意: 上記は例。実DBスキーマに合わせてカラム名・値を修正してください。
```

3. アプリ起動前にマイグレーション/シード処理が走る環境であれば、`data.sql` に追記して起動時に反映させる。そうでない場合は直接DBへINSERTしてテストする。

### 3) バックエンド：おすすめ記事取得APIの確認/修正
1. 既存APIを確認し、**おすすめ（今週の注目）は公開/非公開を問わず全記事を対象**として返す専用の軽量APIを追加する方針で実装を行う。理由: 現行の `GET /api/v1/public/knowledge` は `visibility=public` と `status=published` で絞られており、非公開記事は含まれないため要件を満たさない。

2. 推奨実装（最小変更） — 追加 API とデータ取得ロジック:
  - 新規エンドポイント: `GET /api/v1/public/knowledge/recommended?limit=1`（認証不要）
  - このエンドポイントは **公開/非公開問わず全記事** を対象に `like_count` 降順で上位1件（トップ1）のメタ情報（`id`, `title`, `likeCount`）を返す。本文は含めない。\
  - 実装場所の例: 新規メソッド `KnowledgeRepository.findTopRecommendedArticlesAll(int limit)`（native query で `WHERE k.is_deleted = false` のみ）を追加、`PublicKnowledgeController` にハンドラを追加し `PublicKnowledgeService` で変換して返却する。\

3. 公開記事（画面中央・新着）は既存の `GET /api/v1/public/knowledge` を利用する（`visibility=public` かつ `status=published` のまま）。必要なら公開向けの新着専用パス `GET /api/v1/public/knowledge/new?limit=3` を追加しても良い。

4. 実装手順（概略）:
  - `KnowledgeRepository` に以下の nativeQuery を追加する（例）:

```java
@Query(value = "SELECT k.id, k.title, COUNT(l.id) AS like_count FROM knowledge k LEFT JOIN \"like\" l ON k.id = l.knowledge_id WHERE k.is_deleted = false GROUP BY k.id, k.title ORDER BY like_count DESC LIMIT :limit", nativeQuery = true)
List<Object[]> findTopRecommendedArticlesAll(@Param("limit") int limit);
```

  - `PublicKnowledgeService`（または専用 `PublicRecommendedService`）に変換処理を追加して DTO を返す。
  - `PublicKnowledgeController` に `@GetMapping("/recommended")` を追加して結果を返却する。レスポンスは `[{"id":1001, "title":"...", "likeCount":15}]` のような軽量配列（トップ1）で良い。
  - 単体テスト（Repository のクエリ検証、Service/Controller の MockMvc テスト）を追加する。

5. APIのレスポンス例（JSON）:

```json
{
  "items": [
    { "id": 1001, "title": "公開テスト記事 A", "likeCount": 15 }
  ]
}
```

### 4) フロントエンド実装
1. 対象ファイルを特定する（公開トップ画面のコンポーネント）。たとえば `NaviAI-Front/app/public-home` 配下の該当コンポーネントを編集する。\
2. 静的なタイトル「AIで効率化する記事まとめ」を表示している箇所を削除し、APIから取得した上位記事のタイトルを表示するロジックに置き換える。
3. 実装のポイント:
  - 画面ロード時に `fetch('/api/v1/public/knowledge/recommended?limit=1')` などでおすすめAPIを呼び出す（公開/非公開問わず全記事を対象）。ヘッダ領域はトップ1件を表示する。\
  - 公開記事（画面中央・新着）については既存の `GET /api/v1/public/knowledge` を `?limit=2` で呼び出し、最大2件を表示します（テストデータは3件用意して表示確認）。\
  - レスポンスが空の場合は「データがありません」と表示する。\
  - エラーハンドリング（ネットワーク・JSONパースエラー）を行う。

#### おすすめ記事のクリック挙動（重要）
1. 公開トップ画面からおすすめ記事の本文へは直接遷移しない仕様とする。記事一覧やタイトルは表示するが、本文へのアクセスは認証済みユーザーのみとする。\
2. 実装案:
  - 未認証（または ログイン状態を判定できない）場合、タイトル横または下に `ログインして続きを読む` ボタンを表示する。ボタンを押すとログイン画面に遷移する（ログイン成功後に元の記事詳細へリダイレクトして本文を表示）。\
  - 認証済みユーザーはタイトルをクリックすると記事詳細（本文）に遷移できる。\
3. 実装ポイント:
  - Next.js の認証フローを利用して、未認証時は記事詳細遷移をブロックしログインへ誘導する。\
  - UI上はタイトルはリンク風に見せつつ、クリックハンドラで認証チェックを行い、未認証ならモーダルやボタンでログイン導線を表示するのがユーザビリティ上良い。

フロント実装の擬似コード（概念例）:

```tsx
useEffect(() => {
  async function loadTop() {
    try {
      const r = await fetch('/api/v1/public/knowledge/recommended?limit=1');
      const j = await r.json();
      setTopItems(j.items || []);
    } catch (e) {
      setTopItems([]);
    }
  }
  loadTop();
}, []);

// JSX: topItems.length === 0 -> 'データがありません' を表示
```

1. `NaviAI-Front` 配下で docker compose を起動して Front/Back を立ち上げ、画面表示を確認する。

推奨コマンド:

```bash
cd NaviAI-Front

```

2. ブラウザで公開トップ画面を開き、以下を確認する:
  - ヘッダの「今週の注目」にAPIから取得したトップ1件のタイトルが表示される。中央の新着は公開記事上位2件が表示される。
  - 「データがありません」は表示されない（テストデータあり）。
  - 右上の会員訴求セクションが削除されている。
3. コンテナログで起動エラーが出る場合はログを確認し、原因を修正する（例: ビルドエラー、DBマイグレーション失敗など）。

## テスト／確認手順
- バックエンド: `data.sql` に追加した3件がDBに反映されているかを確認する（SQLでSELECT）。\
- API: 以下をcurl/Postmanで呼び出し、期待したJSONが返るか検証。\
  -- `GET /api/v1/public/knowledge/recommended?limit=1` （おすすめ：全記事を対象、like_count降順、トップ1）\
  -- `GET /api/v1/public/knowledge?limit=2`（公開記事（新着）：公開記事のみ、既存の `GET /api/v1/public/knowledge` を利用、最大2件表示）\
- フロント: ブラウザで公開トップを確認し、上位2件が表示されることを目視で確認。\
- レスポンシブ: モバイル・タブレット表示で崩れがないか簡易チェック。

例: API確認コマンド

```bash
curl -s http://localhost:8080/api/v1/public/knowledge/recommended?limit=1 | jq .
```

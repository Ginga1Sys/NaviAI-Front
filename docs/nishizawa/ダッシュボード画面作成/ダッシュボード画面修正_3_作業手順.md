## **概要**
- **目的**: ダッシュボード（SCR-03）で「おすすめ記事」と「新着記事」が正しく表示されない問題を調査・修正し、管理者用右カラム（承認待ち・週次アクティビティ）の表示も確認する。
- **成果物**: 修正済み画面資産、修正手順書、動作確認手順とスクリーンショット

## **前提条件**
- **リポジトリ**: NaviAI-Front と NaviAI-Back をローカルにクローン済み
- **起動方法**: `NaviAI-Front` フォルダ配下で `docker compose -f docker-compose.dev.yml up --build` が実行できること
- **テストデータ**: `NaviAI-Back/app/src/main/resources/data.sql` にサンプルデータが存在すること

## **参考資料**
- **要件定義**: proj-1sys-ax-2025/docs/00_personal/nishizawa/統合要件定義書.md
- **基本設計（API）**: proj-1sys-ax-2025/docs/00_personal/nishizawa/基本設計書/基本設計_API.md
- **フロント構成**: NaviAI-Front/docs/nishizawa/NaviAI-Front_ディレクトリ構成.md

## **想定される原因**
- フロントエンド側の表示ロジック（APIレスポンスのハンドリング、条件分岐、コンポーネントのprops渡しなど）
- テストデータ不足またはデータ形式の不一致
- 管理者フラグや権限判定ロジックの不備

## **実装方針**
- 再現手順で問題を確認し、ブラウザ開発者ツールでAPIレスポンスを確認する。
- フロント側でAPI呼び出し元（service/lib）→画面（components）→表示ロジックの順に追跡し、原因箇所を最小修正する。
- テストデータに不足があればバックエンドのdata.sqlを一時的に修正して動作確認する（必要なら差し戻しを注記）。

## **詳細な実装手順**
1. 事前準備
   - ローカルの `NaviAI-Front` と `NaviAI-Back` を最新ブランチに切り替え、依存関係をインストールする。
   - `NaviAI-Back` を起動してテストデータがDBに投入されていることを確認する。
   - テストデータの確認:
   - `NaviAI-Back/app/src/main/resources/data.sql` の内容を確認し、サンプル記事（おすすめ、公開状態、作成日時等）が含まれていることを確認する。
   - DBコンソール（H2など）または直接SQLで以下を実行してサンプルデータを確認する:

```sql
SELECT id, title, status, is_recommended, created_at FROM articles ORDER BY created_at DESC LIMIT 5;
```

   - APIで直接確認する場合は、バックエンド起動後に次のようなcurlで確認する:

```bash
curl -s http://localhost:8080/api/v1/articles?sort=new | jq '.' > sample_articles.json
curl -s http://localhost:8080/api/v1/summary/recommended | jq '.' > sample_recommended.json
```

   - データが不足している場合は `data.sql` にサンプル記事を追加してからバックエンドを再起動し、再度確認する（変更はレビューで説明すること）。

2. 画面の再現確認
   - `NaviAI-Front` を起動して `http://localhost:3000/dashboard` を開く。
   - 管理者ユーザーでログインし、右カラム（承認待ち・週次アクティビティ）とおすすめ/新着記事の表示状況を確認する。

3. ネットワーク/APIレスポンス確認
   - ブラウザの開発者ツールの Network タブを開き、ダッシュボード表示時に呼ばれるAPI（おすすめ記事、新着記事、ユーザー情報、管理者用データ等）を特定する。
   - 各APIのレスポンスボディを確認し、期待されるデータが返っているかをチェックする。
   - 例: `GET /api/v1/summary/recommended`、`GET /api/v1/articles?sort=new`（実際のエンドポイント名は基本設計を参照）

4. フロント側の診断（ソース追跡）
   - API呼び出し用ユーティリティ（`NaviAI-Front/app/lib` や `services`）を探し、レスポンスが正しくパースされているか確認。
   - ダッシュボード関連コンポーネント（`NaviAI-Front/app/dashboard` または近接するコンポーネント）を開き、propsやstateによる表示条件を確認する。
   - 「記事がありません。」を表示している条件分岐（件数判定や null チェック）が誤っていないか確認する。

5. 修正方針（発見時）
   - APIレスポンスが期待通り → 表示ロジックのバグ。条件分岐やmap処理の修正を行う。
   - APIレスポンスが不足している → バックエンドのdata.sqlやエンドポイント実装を確認し、テストデータを追加またはAPI実装の不備を報告/修正。
   - 管理者表示が出ない場合 → ユーザー情報取得ロジックで管理者フラグが正しく判定されているか確認。必要ならログインレスポンスやユーザーDTOを確認。

6. コード修正とレビュー
   - 最小の変更で問題を修正する（既存の命名規則やコードスタイルに合わせる）。
   - 修正後、ユニットテスト／既存のフロントテスト（あれば）を実行する。

7. 動作確認
   - `NaviAI-Front` を再ビルドして `http://localhost:3000/dashboard` で表示を確認。
   - ブラウザでおすすめ記事1件、新着記事最大3件が表示されることを確認する。
   - 管理者でログインした際に右カラム（承認待ち・週次アクティビティ）が表示されることを確認する。
   - スクリーンショットを取得して成果物として保存する。

8. ドキュメント更新
   - 修正内容と差分（変更ファイル一覧）を `NaviAI-Front/docs/nishizawa/NaviAI-Front_ディレクトリ構成.md` に追記する。

## **チェックリスト**
- **再現確認**: ダッシュボードで「記事がありません。」が出ることを確認済み
- **API確認**: 関連APIのレスポンスを取得・保存（JSON）
- **修正実施**: フロント修正またはバックエンドのテストデータ修正を実施
- **動作確認**: おすすめ/新着記事と管理者右カラムの表示確認
- **スクリーンショット**: 修正前後のスクリーンショットを保存
- **ドキュメント**: ディレクトリ構成に修正履歴を追記

# 検索結果画面（SCR-04）修正 作業手順書

## 概要
- ダッシュボードの「新着記事」「おすすめ記事」および共通ヘッダーの検索ワードから検索結果画面へ遷移した際に、正しく検索結果が表示されない不具合を調査・修正する。

## 前提条件
- 開発環境（Dockerまたはローカル起動）が動作していること
- フロントエンドリポジトリ: `NaviAI-Front`（feature/dashboard_searchlist ブランチ想定）
- バックエンドリポジトリ: `NaviAI-Back`（feature/search_result_list ブランチ想定）
- 動作確認用データ: `NaviAI-Back/app/src/main/resources/data.sql`

- 基本設計（API）: [proj-1sys-ax-2025/docs/00_personal/nishizawa/基本設計書/基本設計_API.md](proj-1sys-ax-2025/docs/00_personal/nishizawa/基本設計書/基本設計_API.md)

## 成果物（想定）
- 修正済みフロントエンド画面: `NaviAI-Front/app/search_list`（または該当コンポーネント）
- 必要に応じたバックエンド修正: `NaviAI-Back/app/src/main/...` の検索API実装
- 更新ドキュメント: `NaviAI-Front/docs/nishizawa/NaviAI-Front_ディレクトリ構成.md`

## 実装方針
- まず再現手順を確立し、フロントエンドとバックエンドのどちらが原因か切り分ける。
- フロントエンドで遷移時のクエリ/パラメータの受け渡しと、API呼び出しのリクエスト内容を確認する。
- APIが期待するパラメータとフロントエンドが送るパラメータが一致しない場合、双方の最小修正で整合させる。

## 詳細な実装手順
1. 環境準備
   - リポジトリを最新ブランチへ切替（必要なら作業用ブランチ作成）
     ```bash
     cd d:\AI駆動開発デモ\NaviAI-Front
     git checkout feature/dashboard_searchlist
     cd ..\NaviAI-Back
     git checkout feature/search_result_list
     ```
   - 開発環境を起動（フロント中心に検証する場合）
     ```bash
     cd d:\AI駆動開発デモ\NaviAI-Front
     docker compose -f docker-compose.dev.yml up --build
     ```
   - バックエンドが別コンテナの場合は `NaviAI-Back` も起動する。

2. 再現手順の明確化
   - ブラウザで `http://localhost:3000` を開き、ダッシュボードの「新着記事」「おすすめ記事」ボタンを押下して `http://localhost:3000/search_list` へ遷移する手順を録る。
   - 共通ヘッダーの検索入力にワードを入れて検索したときの挙動を確認する。
   - ブラウザ開発者ツールの Network タブで、検索画面遷移時に実行される API リクエスト（URL、クエリパラメータ、HTTPメソッド、レスポンス）を記録する。

3. フロントエンドの調査（原因切り分け）
   - 確認対象ファイル（例）
     - `NaviAI-Front/app/search_list`（一覧表示コンポーネント）
     - `NaviAI-Front/app/dashboard`（ダッシュボードの遷移処理）
     - `NaviAI-Front/app/components/header`（検索入力からの遷移処理）
   - 見るべきポイント
     - ダッシュボードから遷移する際に検索語やフィルタをクエリに付与しているか（例: `?q=xxx`）
     - 検索画面側でクエリパラメータを正しく読んでAPI呼び出しに渡しているか
     - API呼び出しのエラーハンドリングや空配列判定ロジックに問題がないか（「検索結果がありません」の表示条件）

4. バックエンドの確認
   - 実行するAPIエンドポイントとクエリパラメータ仕様を確認する（`基本設計_API.md` の該当箇所）
   - ローカルでAPIを叩いて期待する結果が返るか確認する（例）
     ```bash
     curl -v "http://localhost:8080/api/v1/articles?query=テストワード"
     ```
   - レスポンスの JSON 構造がフロントエンドの期待と合っているか確認する（配列・フィールド名など）
   - 必要であれば、バックエンドで検索条件の受信/パース、またはクエリ側の実装を修正する。

5. 修正の実施（フロントエンド優先）
   - フロントエンド側でクエリパラメータの受け渡しミスを修正する（リンク生成、router.push、useEffect の呼び出しタイミング等）
   - API 呼び出しの URL/パラメータ組み立て、ヘッダー、タイミング（初回表示時に fetch するか等）を修正する
   - 変更点を小さくして単体で動作確認を行う

6. 結合検証
   - フロント・バック両方を立ち上げ、ダッシュボード→検索結果、ヘッダー検索の両方で正しく結果が表示されることを確認する
   - Network タブやバックエンドログでAPIのリクエスト/レスポンスを確認する

7. ドキュメント更新
   - `NaviAI-Front/docs/nishizawa/NaviAI-Front_ディレクトリ構成.md` を必要箇所更新
   - `NaviAI-Back/docs/nishizawa/NaviAI-Back_ディレクトリ構成.md` を必要箇所更新
   - 本手順書（このファイル）を所定フォルダへ保存済みであることを確認

8. レビューとマージ準備
   - 変更内容を小さなコミットに分け、プルリクエストを作成する
   - レビュー担当者に動作確認手順（再現手順と確認用URL）を通知する

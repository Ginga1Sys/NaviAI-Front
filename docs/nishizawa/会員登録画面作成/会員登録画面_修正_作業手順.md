# 会員登録画面_修正_作業手順

## **概要**
- **目的**: フロントエンドとバックエンド双方の修正により、ユーザー登録時の「メール送信完了」「登録完了」「登録失敗」の画面遷移と表示を実装し、動作確認手順を明記する。

## **前提条件**
- **環境**: 開発マシンに Docker/Docker Compose が導入されており、リポジトリがローカルにクローン済みであること。
- **起動コマンド**: フロントエンドを起動して動作確認する際は、`NaviAI-Front` ディレクトリで次を実行する: `docker compose -f docker-compose.dev.yml up --build`

## **成果物（想定）**
- **フロントエンド**:
  - `会員登録送信完了` 画面（メール送信成功メッセージ）
  - `登録完了` 画面（メール内URLアクセスで表示）
  - `登録失敗` 画面（エラー時表示）
  - 既存の `login` / `register` とデザイン整合する CSS 資産
- **バックエンド**:
  - ユーザー登録 API のレスポンス/リダイレクト仕様の調整（メール内リンクアクセス時に登録完了/失敗用 URL にリダイレクト）
- **ドキュメント**:
  - `NaviAI-Front/docs/nishizawa/会員登録画面作成/会員登録画面_修正_作業手順.md`（このファイル）
  - 更新済みの `NaviAI-Front` と `NaviAI-Back` のディレクトリ構成ドキュメント

## **実装方針（方針と注意点）**
- **API 互換性を優先**: 既存のクライアント/テストに影響が出ないよう、まずバックエンドの現在のレスポンスを調査。後方互換が崩れる変更は避けるか、影響範囲を限定して調整する。
- **シンプルな画面追加**: 既存の Next.js 構成（`app/` 以下）にページを追加し、既存の `common` スタイルを利用してデザインを揃える。
- **リダイレクト実装**: 登録確認（メール内リンク）処理はバックエンドで登録処理を行い、完了/失敗に応じてフロントエンドの固定 URL にリダイレクトする（例: `/register/complete` または `/register/failed`）。
- **レビュー必須**: ファイル追加・修正後は必ずコードレビューを受ける（プロジェクトのレビュールールに従う）。

## **詳細な実装手順**

1. 事前確認（担当者: 開発者）
   - 参照資料を開き要件を確認する。
   - 影響範囲（関連 API、既存のテスト、他画面）を洗い出す。

2. バックエンド API の挙動確認（担当者: バックエンド担当）
   - 現行のユーザー登録 API の実装を確認する（`NaviAI-Back/app/src/main/...` 配下）。
   - 登録 API が返すレスポンス（HTTP ステータス、JSON ボディ、あるいはリダイレクト挙動）を確認する。
   - テスト環境でユーザー登録を実行し、返却内容とメール送信のワークフローを確認する。

3. バックエンド修正（必要な場合）（担当者: バックエンド担当）
   - 仕様案:
     - POST `/api/v1/auth/register` は従来通り登録処理を行い、成功時は 200 系の応答とともにメール送信をトリガーする（既存仕様を維持）。
     - メール内の確認リンクはバックエンドの確認エンドポイント（例: `/api/v1/auth/confirm?token=...`）に向け、確認エンドポイントが処理後にフロントエンドの完了/失敗ページへ HTTP リダイレクトする。例: 302 -> `/register/complete?email=...` または `/register/failed?reason=...`。
   - 実装変更点は最小にして、必ずユニットテスト・統合テストを追加・更新する。

4. フロントエンド実装（担当者: フロントエンド担当）
   - 追加ページ（Next.js app router 想定）:
     - `/register/mail-sent` : 登録ボタン押下後に表示する「メール送信完了」画面（ユーザーにメール確認を促す文言とログインへ戻るリンク）
     - `/register/complete` : メール内リンクで遷移される「登録完了」画面（ログイン画面へのリンク）
     - `/register/failed` : 「登録失敗」画面（再登録リンクを表示）
   - 各ページは既存の `login`/`register` の共通ヘッダーや CSS を流用してデザインを統一する。
   - 必要なコンポーネントやスタイルは `NaviAI-Front/app/components` と `NaviAI-Front/app/styles` に配置する。

5. フロント ↔ バック連携の確認（担当者: 両者）
   - 登録フォームから POST を送信し、フロントは成功時に `/register/mail-sent` を表示する（バックエンドが 200 を返した場合）。
   - メール内の確認リンクをブラウザで開くと、バックエンドの確認処理が実行され、成功/失敗に応じてフロントの `/register/complete` または `/register/failed` にリダイレクトされることを確認する。

6. ディレクトリ構成ドキュメント更新（担当者: 実装者）
   - フロント: `NaviAI-Front/docs/nishizawa/NaviAI-Front_ディレクトリ構成.md` に追加したページ/コンポーネントの場所を追記する。
   - バックエンド: `NaviAI-Back/docs/nishizawa/NaviAI-Back_ディレクトリ構成.md` に、変更したコントローラやエンドポイントを追記する。

7. ビルドと動作確認（担当者: 実装者）
   - フロントエンドの起動（`NaviAI-Front` フォルダ）:

```powershell
cd NaviAI-Front
docker compose -f docker-compose.dev.yml up --build
```

   - ローカルでメール送信の代替（本番メール不可時）は、バックエンドのメール送信をログ出力やダミー SMTP に切り替えて確認する。

8. テスト・レビュー・マージ
   - ユニット/統合テストを追加し、CI（存在する場合）でテストが通ることを確認する。
   - 実装内容はプルリクエストを作成し、レビューを受ける（レビューで問題なければマージ）。

## **チェックリスト**
- **設計確認**: 参照資料と要件の整合を確認した。
- **API 互換性**: 既存クライアント・テストへの影響範囲を精査した。
- **バックエンド**: 確認エンドポイントが完了/失敗で適切にリダイレクトすることを実装/確認した。
- **フロントエンド**: 三つの画面（`mail-sent` / `complete` / `failed`）を追加し、デザインが既存に一致することを確認した。
- **ドキュメント**: 両方のディレクトリ構成ドキュメントを更新した。
- **動作確認**: `docker compose -f docker-compose.dev.yml up --build` で動作確認した。
- **レビュー完了**: レビューで承認を得た。

## **補足（確認事項・留意点）**
- メール内リンクのパラメータに機密情報を含めない（トークンは一意な確認トークンを使用する）。
- UX: メール送信完了画面には「メールが届かない場合の案内（迷惑メールフォルダ確認など）」を記載する。
- テスト: E2E テスト（Playwright / Jest）で登録フローの簡易シナリオを追加することを推奨する。
